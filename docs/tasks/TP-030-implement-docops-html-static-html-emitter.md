## Goal

Ship `docops html` — a one-shot CLI subcommand that reads the current repository's DocOps documents through the read layer and emits a directory of self-contained HTML files covering all CTX, ADR, and TP documents plus a STATE overview page.

End state: running `docops html` in any DocOps-enabled repo produces a browsable `docs/.html/` directory that can be opened locally or deployed to any static host.

## Acceptance

### Output structure

`docops html` produces the following directory tree (default output: `docs/.html/`):

```
docs/.html/
  index.html          — STATE overview + per-kind listings with links
  state.html          — full STATE.md rendered as HTML
  ctx/
    CTX-001.html      — per-context detail pages
    CTX-002.html
    ...
  adr/
    ADR-0001.html     — per-decision detail pages
    ADR-0002.html
    ...
  task/
    TP-001.html       — per-task detail pages
    TP-002.html
    ...
```

### Index page (`index.html`)

- Renders STATE.md counts, needs-attention, and active-work sections as structured HTML.
- Per-kind tables: CTX, ADR, TP — each row shows ID, title, status (with color badge), last-touched date.
- Tables are hyperlinked to their detail pages.
- Navigation bar at top: Home | Context | Decisions | Tasks.

### Detail pages (`ctx/{ID}.html`, `adr/{ID}.html`, `task/{ID}.html`)

- **Frontmatter table**: every frontmatter field rendered as a key-value HTML table. Date fields formatted human-readably. Arrays rendered as comma-separated links where the referenced doc exists.
- **Markdown body**: rendered via goldmark with GFM extensions (tables, autolinks, heading IDs, strikethrough).
- **Graph sidebar**: reverse-edge links showing what references this document (computed from the index).
  - CTX: which tasks require it, which ADRs relate to it.
  - ADR: which tasks require it, which ADRs supersede/relate to it, whether it has amendments.
  - TP: its `requires` and `depends_on` links rendered as clickable references.
- **Breadcrumb**: Home > Kind > ID
- **Footer**: "Generated by docops v{version} on {timestamp}"

### Command flags

| Flag | Default | Description |
|---|---|---|
| `--output` / `-o` | `docs/.html` | Output directory (created if absent; contents replaced if present) |
| `--base-url` | (empty, relative links) | Base URL prefix for all internal links (for hosting behind a path prefix) |
| `--json` | off | Emit `{ "files_written": N, "output_dir": "..." }` to stdout, suppress human output |

Exit codes: 0 success, 2 invalid flags, 1 runtime error (no docs found, permission denied, etc.)

### Go implementation

- New package: `internal/htmlviewer/` containing:
  - `renderer.go` — shared rendering logic (goldmark config, template functions, link resolvers).
  - `templates.go` — Go `html/template` definitions for index, detail, state pages.
  - `css.go` or `css/` via `embed.FS` — embedded CSS theme.
- New command file: `cmd/docops/cmd_html.go`
- `cmd_html.go` calls `internal/htmlviewer.Render(config)` which:
  1. Loads docs via `internal/index.BuildIndex()` (same path as `docops index`).
  2. Iterates each indexed document, renders to HTML using templates.
  3. Writes files to the output directory using `os.MkdirAll` + `os.WriteFile` (atomic per-file).
- Goldmark configuration:
  - Extensions: `extension.Table`, `extension.Linkify`, `extension.Strikethrough`, `extension.Typographer`.
  - Renderer: `goldmark.DefaultRenderer()` with heading ID support.
  - No syntax highlighting in v1 (plain monospace code blocks).

### Data sourcing

- All document data comes from `internal/index.IndexedDoc` structs — the same types returned by `docops get`/`docops list`.
- The viewer **never** reads `docs/.index.json` directly. It calls the in-memory index built by `internal/index.BuildIndex()`.
- This honors ADR-0018: CLI read commands are the query API.

### Tests

- Unit tests for:
  - Goldmark rendering of markdown fixtures (GFM tables, autolinks, headings with IDs).
  - Template rendering of frontmatter tables (arrays, dates, empty fields).
  - Link generation with and without `--base-url`.
  - `--json` output format.
- Integration test:
  - Create a temp directory with sample CTX/ADR/TP docs.
  - Run `docops html --output <temp>`.
  - Assert all expected files exist and contain valid HTML (opening/closing tags).
  - Assert no external resource references (no `href="http"` pointing outside local pages).
- Negative tests:
  - Running `docops html` in a non-DocOps repo (no `docs/` directory) — exit 1 with clear error.
  - `--output` pointing to a read-only path — exit 1 with permission error.
  - Empty docs directory (valid structure but zero documents) — still produces `index.html` with "No documents" message.

## Notes

- The `docs/.html/` directory should be added to `.gitignore` by default. It is a generated artifact equivalent to build output. Users may opt in to committing it for GitHub Pages / Netlify deployment.
- If amendments (ADR-0025/TP-026) have not yet shipped, the viewer simply does not render amendment data. When TP-026 lands, `internal/index.IndexedDoc` grows an `Amendments` field and the ADR detail template renders it without template changes (the read layer handles it).
- Consider adding a `.nojekyll` file to the output directory so GitHub Pages serves it correctly without Jekyll processing.
- The `--base-url` flag should both set `<base href>` in the HTML head and rewrite all internal `<a href>` links. This ensures static hosting behind a path prefix (e.g. `https://org.github.io/repo/`) works out of the box.

## Out of scope

- JavaScript or interactivity of any kind (search, filtering, collapsible sections).
- Theme customization (see TP-032).
- Auto-refresh or live preview (see TP-031).
- PDF or other format export.
- Authentication, authorization, or multi-repo views.
