## Goal

Design and implement the default CSS theme for the docops HTML viewer (`docops html` and `docops serve`). The theme must look professional, render markdown faithfully, work offline, and be fully embedded in the compiled binary.

End state: every HTML page emitted by `docops html` renders with a polished, consistent theme that requires zero external resources and looks good at all viewport widths.

## Acceptance

### Design system

- **Font stack**: system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif. Monospace: "SF Mono", "Fira Code", "Cascadia Code", Menlo, Consolas, monospace.
- **Base size**: 16px root. Line-height 1.6 for body, 1.4 for tables.
- **Max content width**: 48rem (768px) for body text. Full-width for index/state tables.
- **Color palette**:
  - Background: `#ffffff` (light mode only for v1).
  - Text: `#1a1a2e` (near-black with slight warmth).
  - Muted text: `#6b7280`.
  - Code background: `#f3f4f6`.
  - Per-kind accents (used for badges, headings, sidebar highlight):
    - CTX: `#2563eb` (blue-600)
    - ADR: `#d97706` (amber-600)
    - TP: `#16a34a` (green-600)
  - Link color: `#2563eb`. Visited: `#7c3aed`.
  - Error/deprecated: `#dc2626`.

### Navigation

- Top nav bar: sticky, white background, subtle bottom border (`#e5e7eb`).
- Nav items: Home | Context | Decisions | Tasks â€” left-aligned, kind-colored on hover.
- Breadcrumb: `Home > ADR > ADR-0027` â€” below nav, muted text, links active.

### Typography

- Heading hierarchy: h1 (2rem, bold), h2 (1.5rem, semibold + bottom border), h3 (1.25rem, semibold), h4 (1rem, bold).
- Inline code: backtick-styled with monospace font, light gray background, 0.15rem rounding.
- Code blocks (`pre > code`): full-width, monospace, left-aligned, no overflow hidden (horizontal scroll), faint border.
- Blockquotes: left border in muted color, italic text.

### Tables (frontmatter metadata + GFM tables)

- Bordered: `1px solid #e5e7eb`. Collapsed borders.
- Header row: light gray background (`#f9fafb`), bold text, left-aligned.
- Zebra striping on even rows (`#f9fafb`).
- Key column (frontmatter tables): `font-weight: 600`, 30% width.
- Value column: 70% width, arrays rendered as comma-separated.

- Content: `Generated by docops v{version} on {date}`

### Accessibility

- Color contrast ratios meet WCAG AA (4.5:1 for normal text, 3:1 for large text).
- All interactive elements are keyboard-navigable.
- Status badges use `aria-label` with the status text.
- Headings use logical nesting (h1 â†’ h2 â†’ h3, never skipping levels).

### CSS embedding

- All CSS is compiled into a Go `embed.FS` or string constant in `internal/htmlviewer/css/`.
- Template references: `<style>{{ .CSS }}</style>` in the `<head>` of every page.
- No external stylesheet links. No `@import`. No CDNs.
- Total CSS budget: under 8 KB minified (before gzip).

### Dark mode

- **Out of scope for v1.** The CSS variables and structure should be designed so dark mode is easy to add in a follow-up task (CSS custom properties for all colors), but the implementation is deferred.
- All color values in the CSS should use CSS custom properties (`--color-text`, `--color-bg`, etc.) so a future `prefers-color-scheme: dark` or `--theme=dark` toggle swaps the palette with minimal changes.

## Notes

- Use CSS custom properties (variables) for every color, spacing, and font value. This makes the theme customizable-in-principle (future `--theme` flag) and dark-mode-ready.
- The `.nojekyll` file added by TP-030 ensures GitHub Pages doesn't process the HTML. The CSS should not rely on any Jekyll-specific features.
- Prefer `rem` units over `px` for spacing, `em` for font-relative sizes. Use `px` only for borders and shims.
- Test rendering in: Firefox, Chrome, Safari. Mobile Safari has specific quirks with `position: sticky` on flex children â€” add `will-change: transform` to the nav bar.

## Out of scope

- Dark mode implementation (structure prepared, implementation deferred).
- Theme customization UI or CLI flag (future enhancement).
- Print stylesheet (can be added in a follow-up).
- Syntax highlighting for code blocks (requires a JS library or server-side highlighting â€” too heavy for v1).
