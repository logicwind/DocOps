## Resolved

### A. Session Storage — RESOLVED

**Decision:** Host-mounted directory with `shift=true`. Sessions survive container restart. zundd reads JSONL directly for status/monitoring.

**Evidence:** POC 02 (host mounts), POC 03 (Pi in Incus), POC 11 (session lifecycle). All validated.

### B. User-to-Agent Interaction — RESOLVED

**Decision:** zundd REST API is the single control plane. `POST /v1/agents/:name/message` with SSE streaming. CLI, console, scripts — all equal API consumers. No `zund shell` escape hatch needed for v0.3.

**Evidence:** POC 04 (RPC proxy, <1ms overhead), POC 10B (custom web UI works against the API).

### C. Ephemeral Agents — RESOLVED

**Decision:** `incus copy --ephemeral` from template container. Auto-deletes on stop. zundd captures run logs. Opt-in session persistence via host mount for agents that need history.

**Evidence:** POC 06 (228ms trigger-to-ready, 3 concurrent clones in 447ms).

### D. Agent Memory — RESOLVED

**Decision:** `@mastra/memory` + `@mastra/libsql` standalone in Bun. Zero external infra (SQLite). Scoped to agent/team/fleet. Semantic recall requires embedder API key. Pi extension exposes `memory_save/search/list` tools.

**Evidence:** POC 05 (Mastra works standalone), POC 09 (Pi integration, 54 tests pass).

### H. Pi Built-in Tool Selection from YAML — RESOLVED

**Decision:** `tools?: string[]` on `AgentResource`, plumbed through
`ExtensionConfig` to a `pi.setActiveTools()` call in the generated extension's
`session_start` hook. `undefined` (omitted) inherits Pi's defaults; `[]`
disables all built-ins; non-empty array is the exact allowlist. Zund's own
extension tools (memory, working-memory, fleet status, load_skill) are always
available regardless. Shipped in commit 3623b37.

**Evidence:** `packages/daemon/src/pi/extension.ts`, `fleet/types.ts::AgentResource`,
`test/unit/extension.test.ts` per-agent tools cases.

### J. File/Artifact Support — RESOLVED

**Decision:** Content-addressed local storage (`LocalArtifactStore`, SHA-256 IDs, `~/.zund/data/artifacts/blobs/`). Skills emit files explicitly via the `zund_emit_artifact` Pi tool — no filesystem scanning. Policy (`maxSizeMb`, `retention`, `allowedKinds`) lives under `kind: defaults { agent: { artifacts: {...} } }` and deep-merges per-agent via the existing defaults path. Console renders inline by full MIME type (audio/image/video/PDF/markdown/HTML); office formats fall through to a labeled download link. TTL sweeper runs on a 30-minute timer; blobs are reference-counted before unlink (dedup safety). Storage interface is pluggable — S3/MinIO drops in later without touching callers.

**Evidence:** `docs/slice-7-plan.md` (design + sub-slices), `.claude/plans/peppy-scribbling-origami.md` (execution detail).

## Open

### G. Per-Agent Memory DB Storage Strategy

**Context:** Current memory lives in one `~/.zund/data/memory.db` with scope columns — all agents share the file. Question raised: should each agent have its own DB (mounted into its container), with a separate shared DB for `team:` / `fleet:` scopes?

**Three shapes on the table:**

1. **Logical split, daemon-mediated.** `~/.zund/data/memory/<agent>.db` + `~/.zund/data/memory/_shared.db`. Agents still curl the daemon. Zero container changes. Gets isolation, easy reset, cheap backups. DB does not "travel with" the container.
2. **Bind-mount per-agent DB into container, daemon still the writer.** Mount `~/.zund/data/memory/<agent>.db` → `/root/.zund/memory.db`. Extension can read directly via `bun:sqlite` (zero-latency recall); writes still go via HTTP so the daemon stays the single writer and can emit SSE events. Two read paths to keep consistent.
3. **Full local ownership, agent writes directly.** Multi-writer SQLite over a bind-mount. WAL + POSIX locks make this work in theory, but `-wal`/`-shm` files, userns UID mapping, hard-kill corruption, and harness-UI-vs-container races are new edge cases. Also loses the daemon's ability to validate/quota/emit before a write lands.

**Shared team/fleet DB:** in all three, `_shared.db` stays daemon-mediated. Multi-container writers to one file is the same problem as #3 but with N containers. Keep HTTP-mediated regardless.

**Key questions to resolve:**
- *Why in the container?* Debuggability (`sqlite3 /root/.zund/memory.db`) → #1 + a debug endpoint is enough. Recall latency → #2. Self-contained snapshottable agent → #3.
- Does `DELETE /v1/fleet` delete memory? Today it doesn't. With per-agent files the default becomes load-bearing.
- Schema migration: one DB → one migration. N DBs → daemon migrates each on open.
- Cross-agent fact search: today a single query with scope filter. Per-agent files make this a fan-out — is it needed, or is `team:`/`fleet:` sufficient?

**Current thinking:** #1 is the low-risk operational win (isolation, backup, reset) with no concurrency risk; defer #2 until a real latency problem appears; #3 only if snapshottability becomes a product requirement. Parked until after integration tests land.

**Status:** Open — revisit after memory/skills integration tests prove the current shape works end-to-end.

### I. CLI follow-ups (deferred from Slice 7)

Slice 7 shipped `@zund/cli` with a feature-complete command surface, but
four pieces were intentionally deferred. Each is small and unblocked.

1. **Bundle sops + age binaries.** Wave 5's `packages/daemon/src/secrets/`
   wrapper looks for `sops` in `~/.zund/bin/` first, but nothing puts it
   there yet. Two paths: (a) npm `postinstall` script that downloads the
   right platform binary from the SOPS / age GitHub releases into
   `~/.zund/bin/`; (b) Homebrew formula `resource` blocks (already
   drafted in `packages/cli/Formula/zund.rb`) that stage them in
   `libexec` and `caveats` walks the user through one-time symlinks.
   Need both because npm and brew are independent install paths.
   Acceptance: `zund secret set FOO bar` works on a fresh machine with
   only `npm i -g @zund/cli` or `brew install zund`, no manual sops.

2. **GitHub Release pipeline.** A workflow that on tag push runs
   `packages/cli/scripts/build-binaries.sh`, uploads the four binaries
   (darwin-arm64/x64, linux-arm64/x64) as Release assets, computes
   SHA256s, and patches `Formula/zund.rb` in the `homebrew-zund` tap
   repo (or this repo if we don't split the tap yet). Until this exists
   the formula's SHA256 placeholders are blockers for the first brew
   install.

3. **TUI for `zund chat`.** Plain ANSI streaming ships today via
   `node:readline`. A real TUI (Ink or blessed) would add scrollback,
   collapsible tool-call blocks, and split panes. Defer until a real
   user complaint surfaces — the 80-line readline implementation is
   meeting the bar for now.

4. **`zund agent create` / `zund agent delete` wizards.** The current
   workflow is "edit YAML by hand, then `zund fleet apply`." A wizard
   that scaffolds an agent file from prompts would lower the new-user
   bar but would also duplicate the `init` wizard's work. Worth doing
   *after* we see what new users actually type wrong.

5. **`zund secret audit` command.** Static-analysis command that flags
   (a) plaintext-looking API keys in `fleet/**/*.yaml` via regex
   (`sk-`, `AKIA`, `ghp_`, `xox[baprs]-`, etc.); (b) agent secret
   mappings whose target name is missing from the decrypted vault;
   (c) role/skill `secrets.required` entries that aren't mapped by any
   active agent; (d) precedence shadowing where a plaintext value is
   silently overridden by a ref. Borrowed from OpenClaw's `openclaw
   secrets audit` pattern. Pure static analysis, ~60 LOC, runs in CI.
   Unblocked — pick up after this wire-through lands.

6. **`zund apply` no-diff case skips secret re-resolution.** When
   the incoming YAML matches the running fleet state exactly, the
   executor returns early with an empty plan and `resolveFleetSecrets`
   never runs. Consequence: rotating a vault secret and re-running
   `zund apply` is a silent no-op — the containers keep the stale
   plaintext they were booted with. The eager-snapshot semantics
   documented in `docs/secret-management.md §Resolution` promise
   "resolve on every apply", but the current implementation only
   delivers "resolve on every apply *that has work to do*." Two
   options: (a) compare the resolved snapshot against the last-applied
   snapshot and restart any agent whose env diverges; (b) add a
   dedicated `zund secret rotate` that forces a restart of agents that
   reference the rotated secret name. (b) is less surprising and
   avoids snapshot persistence. Caught during Slice 7 end-to-end
   verification in `samples/secrets-test/`.

**Status:** Open — pick up after the v0.3 milestone settles.

### K. Multi-User Agent Sessions — Parallel Conversations on One Agent

**Context:** Today an "agent" is one `pi --mode rpc` process inside one
container, with one live session at a time. Pi exposes `new_session`,
`switch_session`, and `fork` RPC commands, but pi itself is a **personal**
coding agent (`@mariozechner/pi-coding-agent`) — its session model assumes
one human at the keyboard. Zund currently ships `POST /v1/agents/:name/session/new`
(rotates the live slot) and a matching CLI / console button, verified working
end-to-end against `gemma4:e4b` on 2026-04-16 — but that's still single-session.

The open question: **what happens when multiple users, tabs, or automations
need to converse with the same agent concurrently?** Each with their own
thread, without stepping on each other.

**Pi's hard constraint:**
Pi has exactly one live session per `pi --mode rpc` process. Commands are
serialized on a single JSONL stdin/stdout pipe (see
`packages/daemon/src/pi/rpc.ts::AgentRpcSession` — `sendCommand` is
single-inflight via `pendingResolve`/`pendingReject`). You cannot make one
pi process serve two conversations at once. Anything that looks "parallel"
on top of a single pi is really serialized-plus-switching.

**Four architectural shapes on the table:**

1. **Accept single-user agents (recommended default).** An agent is one
   user's workspace. Multi-user means multi-agent — the fleet defines roles
   (poet, reviewer); each user gets their own instance of the role. Shared
   skills, shared memory (via `team:` / `fleet:` scopes in the memory DB),
   isolated sessions. Matches pi's design, matches how humans actually
   think about threads, and the team-knowledge story already works through
   the memory scopes resolved in decision D.

   **Trade-off:** the fleet grows linearly with users. Container cost per
   user. Fine for ≤dozens, awkward for thousands.

2. **One pi process per concurrent session inside the same container.**
   Spawn N `incus exec ... pi --mode rpc` processes inside one agent
   container. Each handles one session, daemon routes messages by
   session-id → pi-pid. Pi doesn't know about siblings. Daemon tracks a
   `Map<sessionId, AgentRpcSession>` per agent instead of one session.

   **Trade-off:** memory. Each pi loads its own model client, skill cache,
   context window. On an 8GB Pi 5 you'd cap around 10–20 concurrent pi
   instances per container. Dev boxes handle more. Container-level
   failures still shared.

3. **One ephemeral container per session.** Each active session spins up
   its own container via the existing `--ephemeral` path (decision C,
   228ms trigger-to-ready from POC 06). True isolation, scales horizontally,
   no sibling interference.

   **Trade-off:** cost. Container spin-up is seconds, and sessions don't
   share warm skill caches or working memory. Acceptable for long-running
   jobs (batch reviews, overnight research), painful for chat. Starts
   looking like the commercial cloud story in the project vision.

4. **Daemon-side queue-and-switch with single pi.** One pi per agent. Daemon
   tracks `currentSessionId` per agent. Every message API call accepts
   optional `sessionId`. If `sessionId !== currentSessionId`, issue a
   `switch_session(sessionPath)` to pi first, then the prompt. From the
   outside it looks parallel; under the hood it's serialized.

   **Trade-off:** head-of-line blocking (user B waits for A's 30s turn +
   a switch round-trip), switch thrashing under rapid alternation, and
   shared failure domain (A's hang kills B). Not real isolation — just
   namespacing. And **this requires `switch_session` to actually work on
   pi**, which is still unverified as of this note.

**Key questions to resolve:**

- *What's the actual use case?* Team of devs each chatting with `poet` from
  the console (→ 1)? One person running 5 parallel automation scripts
  against `reviewer` (→ 2 or 4)? Multi-tenant SaaS where users don't know
  each other (→ 3, looks like the commercial cloud vision)?
- *Is `pi switch_session` reliable?* Needs a manual end-to-end test against
  the running fleet before option 4 is even viable. Write a session, rotate,
  write another, switch back, ask about the first session's content — if
  pi recalls it, switch works. If not, option 4 is dead.
- *How does shared memory interact with session isolation?* Memory scopes
  (`team:`, `fleet:`) already solve "shared team knowledge" without
  sharing conversations. Users probably want shared *skills, config, and
  memory*, not shared conversation threads. Worth making that explicit in
  docs so users don't ask the wrong question.
- *Does the API grow a `sessionId` parameter now?* If we build option 1,
  the API stays agent-scoped. If we ever build 2/4, the API needs
  `sessionId` on every message endpoint and the breaking change is easier
  to do early than late. Argues for reserving the field now even if
  defaulting to implicit.
- *Fleet delete semantics under option 2 or 3:* today delete destroys
  containers; with multiple pi processes or ephemeral-per-session, what
  counts as "all agents gone"? Probably: fleet delete = kill all containers
  and all their pi children, regardless of how many sessions were live.

**Current thinking:**

Option 1 as the default. An agent is one user's workspace. Most "shared
agent" use cases are actually "shared skills / shared memory / shared
config," and those are already covered by existing decisions (skills
mounted read-only, memory scopes, fleet YAML). No session-level sharing
required for the base case.

Option 2 as an opt-in flag on `kind: agent` — something like
`concurrency: N` — for the specific case of one user running multiple
automations against the same agent. Daemon manages an `AgentRpcSession`
pool per agent, keyed by `sessionId`. Only builds the complexity when a
concrete need appears.

Option 3 stays parked until the commercial console/cloud work needs it —
per-tenant isolation is the obvious case for ephemeral containers per
session.

Option 4 is **not recommended**. It looks cheap but the head-of-line
blocking and failure coupling make it an anti-pattern dressed as a
feature. Don't build the illusion of parallelism on top of pi — either
accept single-user or spawn real pi processes.

**Status:** Open — needs the "which use case actually exists" answer
before any code moves. Blocker for multi-tab console usage, but not
urgent until someone hits it.

### E. Knowledge Base: Curated Information for Agents

**Context:** Knowledge is user-provided reference material (brand guides, docs, analytics data) — distinct from memory (agent-generated). Agents query it, don't own it.

**Key questions:**
- Same storage backend as memory (SQLite + vector) or separate?
- Ingest pipeline: chunking + embedding strategy?
- Scope: fleet-level, team-level, or agent-level collections?
- Write access: can agents write to knowledge collections?
- How does this surface in YAML config?

**Current thinking:** Same engine as memory, separate namespace. Fleet/team scoped. Agent-writable collections opt-in. Later-phase feature (Slice 6+) but API surface should be designed alongside memory.

**Status:** Open — design when Slice 6 starts.

### F. Secret Management — RESOLVED

**Decision:** SOPS + age under the hood, wrapped in simple `zund secret` commands. Users never touch SOPS or age directly.

**How it works:**
- `zund init` generates an age key pair automatically (`~/.zund/age.key`)
- `zund secret set name value` encrypts via SOPS, stores in `fleet/secrets/keys.yaml`
- `zund apply` decrypts at apply time, injects as container env vars
- Encrypted secrets file is git-safe — committed with fleet config
- Fleet folder is self-contained: config + encrypted secrets

**Secret requirements flow:**
- `role` and `skill` declare `secrets.required` / `secrets.optional` (what env vars they need)
- `agent` maps requirements to secret names: `secrets: { ENV_VAR: secret-name }`
- `defaults` can set default mappings inherited by all agents
- Preview validates: all required secrets mapped AND exist in encrypted store

**Remote server setup:**
- `zund setup` prompts: "Paste your age key" → one paste, one time
- Or env var: `SOPS_AGE_KEY=AGE-SECRET-KEY-1...` for CI/Docker

**Upgrade path (no zundd code changes):**
- age → AWS KMS: edit `.sops.yaml`, run `zund secret rotate`
- age → Vault: same — change `.sops.yaml`, rotate
- SOPS handles all backends; zundd just calls decrypt

**Env vars as fallback:** `secrets: { API_KEY: ${ENV_VAR} }` for quick starts without SOPS.

## Thought Notes

### zundd: Daemon or Server?

**Resolution:** One process, call it a daemon. Precedent: `incusd` serves REST over Unix socket + HTTPS, has a web UI, nobody calls it a "server." zundd listens on Unix socket (local CLI) + TCP (web/remote). If it gets too heavy later, split. Premature splitting = two processes to deploy for zero benefit.
