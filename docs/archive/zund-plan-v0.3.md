## 1. Package Structure

```
packages/
  daemon/        zundd — all business logic, API, Incus client, Pi management
  cli/           zund CLI — thin HTTP client to zundd, zero business logic
  console/       Web UI — agent management, chat, fleet overview
```

Each package has its own `package.json` and `tsconfig.json` (extends root). The daemon is the only package with real logic — CLI and console are presentation layers.

## 2. Fleet Folder

Fleet config is YAML with `kind` + `name` as identity (like K8s). One file or many — your choice.

### Resource Format

Every resource has `kind` and `name`. These together are the unique identity. Optional `label` (display name) and `tags` (general-purpose metadata).

```yaml
kind: agent
name: content-writer
label: "Content Writer"        # optional — display name in UI, defaults to name
tags: [marketing, content, v2] # optional — general-purpose, filterable
model: claude-sonnet-4-5
skills: [post-creator, brand-voice]
lifecycle: persistent
```

### Defaults (reduce repetition)

A `kind: defaults` block sets base values. Resources inherit them, override only what differs. Inspired by CUE's hierarchical defaults (reduced K8s configs by 45%).

```yaml
---
kind: defaults
agent:
  model: claude-sonnet-4-5
  lifecycle: persistent
---
kind: agent
name: content-writer
skills: [post-creator, brand-voice]
# inherits model + lifecycle from defaults
---
kind: agent
name: analytics
model: gpt-4o              # override just this one
skills: [project-access]
```

### Implicit Kind from Folder

When using a directory layout, `kind` is inferred from the folder name. Explicit `kind` always works and takes precedence.

```
fleet/
  agents/
    content-writer.yaml    # kind: agent inferred
    analytics.yaml         # kind: agent inferred
  skills/
    post-creator.yaml      # kind: skill inferred
  teams/
    marketing.yaml         # kind: team inferred
  defaults.yaml            # kind: defaults (explicit, no folder inference)
```

So the minimal agent file in `fleet/agents/` is just:

```yaml
# fleet/agents/content-writer.yaml
name: content-writer
skills: [post-creator, brand-voice]
```

### Flexible Input

```bash
# Single file with everything (--- separators)
zund apply -f fleet.yaml

# Directory (recursive, all .yaml files)
zund apply -f fleet/

# Subdirectory or specific file
zund apply -f fleet/agents/
zund apply -f fleet/agents/content-writer.yaml
```

All equivalent. Folders are convention, not requirement. Duplicate `kind` + `name` = validation error. Dangling references (skill not found) = validation error.

### IDE Support

Zod schemas generate a JSON schema, published as part of the package. Add to your fleet YAML for autocomplete + validation:

```yaml
# yaml-language-server: $schema=https://zund.dev/schema/v0.3.json
kind: agent
name: content-writer
```

Works in VS Code, JetBrains, and any editor with YAML language server support.

### Read-Only Config

zundd never writes to fleet files. Internal state (container IDs, session paths) lives in zundd's own state on disk. Filenames are organizational — identity comes from the YAML content.

### All Resource Kinds

#### `defaults` — base values inherited by other resources

```yaml
kind: defaults
agent:
  model: claude-sonnet-4-5
  lifecycle: persistent
  compute:
    cpu: 2
    memory: 4GB
  artifacts:              # optional — controls file emission from skills
    maxSizeMb: 25
    retention: 7d
    allowedKinds: [audio, image, document]

# Skill registries — available to all skills (later slice, not v0.3 launch)
registries:
  - name: company
    url: https://skills.company.com
  - name: public
    url: https://registry.skills.sh
    cpu: 2
    memory: 4GB
skill:
  # default skill settings if needed
```

#### `role` — reusable agent template (the "class")

Role defines what an agent IS — persona, safety, compute, sandbox. Multiple agents can share the same role. Agent-level config overrides role defaults.

```yaml
kind: role
name: content-writer
label: "Content Writer"
tags: [marketing]
description: Writes social media content in brand voice

# Persona — injected as system prompt context (supports {{variables}})
persona: |
  You are a social media content writer for {{team.name}}.
  You write engaging posts that match the brand voice.

# Default capabilities (agents can add more)
skills: [social-post, brand-voice]

# Compute defaults (agents can override)
compute:
  cpu: 2
  memory: 4GB

# Safety constraints
safety:
  network: restricted            # restricted | open
  bash_blocked:
    - pattern: "rm -rf /"
      reason: "destructive operation"
    - pattern: "curl.*\\|.*sh"
      reason: "remote code execution"
  read_only_paths: [/etc, /var]

# Sandbox — container-level packages
sandbox:
  packages:
    apt: [curl, jq]
    npm: [sharp]
    pip: [pillow]
```

#### `agent` — a running instance of a role

Agent references a role and adds instance-specific config (model, extra skills, mounts). Everything from the role is inherited; agent-level fields override.

```yaml
kind: agent
name: writer-india
label: "India Content Writer"
tags: [marketing, india]

# Role reference — inherits persona, compute, safety, sandbox, skills
role: content-writer

# LLM (instance-specific — different agents can use different models)
model: claude-sonnet-4-5

# Additional skills (merged with role skills)
skills: [brand-voice-india]

# Lifecycle
lifecycle: persistent          # persistent | ephemeral
trigger:                       # only for ephemeral agents
  type: webhook                # webhook | cron | event
  cron: "0 9 * * 1"           # if type: cron
  timeout: 300                 # max run duration in seconds

# Compute override (or inherit from role)
compute:
  cpu: 4                       # override role's cpu: 2

# Extra mounts — optional, power-user only
# (every agent already gets /workspace automatically, see "Managed Data" section)
mounts:
  - name: brand-assets         # mounted at /data/brand-assets in container
    host: /shared/brand-assets # absolute path on server
    readonly: true

# Container
image: zund/base               # override base image (default from zundd config)
```

Role is optional — an agent without `role:` defines everything inline (useful for one-off agents).

#### `skill` — self-contained capability following Agent Skills spec

Skills are self-contained directories with a `SKILL.md` (Agent Skills open standard). Zund manages where they come from and how they're provisioned into containers.

**Local skill (development, early slices):**
```yaml
kind: skill
name: brand-voice
label: "Brand Voice Guide"
tags: [content, brand]
source:
  type: local
  path: ./skills/brand-voice/     # relative to fleet dir, contains SKILL.md
```

**Built-in skill (shipped with Zund):**
```yaml
kind: skill
name: fleet-tools
source:
  type: builtin                   # provided by zundd (memory, fleet status, etc.)
```

**Git skill (pinned version via ref):**
```yaml
kind: skill
name: social-post
label: "Social Post Creator"
tags: [content, social]
source:
  type: git
  repo: https://github.com/company/agent-skills
  path: skills/social-post        # subdirectory within repo
  ref: v2.1.0                     # tag/branch/commit = version
```

**Registry skill (versioned, production — later slice):**
```yaml
kind: skill
name: deploy-to-vercel
label: "Deploy to Vercel"
tags: [deploy, ci]
source: company/deploy-to-vercel@3.0.0    # registry/name@version
# shorthand: deploy-to-vercel@3.0.0       # uses first registry in defaults
```

**Skill directory structure (Agent Skills spec):**
```
skills/brand-voice/
  SKILL.md              # required — YAML frontmatter + instructions
  resources/            # optional — scripts the agent can execute
    analyze-tone.sh
  references/           # optional — reference docs loaded on demand
    brand-guide.md
    tone-examples.md
  assets/               # optional — templates, images, etc.
```

**How zundd provisions skills:**
- `local` → content uploaded with apply, stored on server
- `builtin` → already in zundd, mounted as Pi extension
- `git` → fetched and cached by zundd on apply
- `registry` → fetched from configured registry, cached, version validated

Skills are mounted readonly into containers at `/skills/{name}/`. Pi discovers `SKILL.md` files automatically.

**Preview validates skills:**
```
$ zund apply --preview
  Skills:
    ✓ brand-voice       local ./skills/brand-voice/ (SKILL.md found)
    ✓ fleet-tools       builtin
    ✓ social-post       git company/agent-skills@v2.1.0 (found)
    ✗ broken-skill      registry company/broken@1.0.0 (not found)
```

**Version management (AgentOps workflow, later slice):**
```bash
# Develop locally with source: local
# Publish to registry: zund skill publish ./skills/social-post --registry company --version 2.1.0
# Reference in production: source: company/social-post@2.1.0
# Upgrade: bump version in YAML, preview, apply
```

**Build order:** local + builtin first (Slice 6), git next, registry last.

#### `team` — group of agents with coordination

```yaml
kind: team
name: marketing
label: "Marketing Team"
tags: [content]
orchestrator: team-lead        # agent that coordinates the team
members: [content-writer, analytics, knowledge]
```

#### `workflow` — multi-step orchestrated flow between agents

```yaml
kind: workflow
name: publish-post
label: "Publish Post"
tags: [content, automated]
team: marketing                # which team runs this
steps:
  - name: draft
    agent: content-writer
    prompt: "Write a social post about {{topic}}"
  - name: review
    agent: reviewer
    input: $prev               # output of previous step
  - name: revise
    agent: content-writer
    input: $prev
    condition: review.needs_revision
  - name: publish
    agent: publisher
    input: $draft
```

## 3. Secrets

SOPS + age under the hood, wrapped in simple `zund secret` commands. Users never touch SOPS or age directly.

### User Experience

```bash
# Init generates age key automatically
zund init --fleet-dir ./fleet
# → creates ~/.zund/age.key + fleet/.sops.yaml (user never sees this)

# Set secrets — just name + value
zund secret set anthropic-key sk-ant-abc123
zund secret set linkedin-key li-key-def456
# → encrypted in fleet/secrets/keys.yaml (git-safe)

# List
zund secret list
#   anthropic-key     ✓ set
#   linkedin-key      ✓ set

# Remove
zund secret remove linkedin-key
```

### How Roles, Skills, and Agents Use Secrets

```yaml
# Role declares what it NEEDS
kind: role
name: content-writer
secrets:
  required: [ANTHROPIC_API_KEY]
  optional: [IMAGE_GEN_API_KEY]

# Skill declares what IT needs
kind: skill
name: social-post
secrets:
  required: [LINKEDIN_API_KEY]

# Agent MAPS names to encrypted store entries
kind: agent
name: writer-india
role: content-writer
skills: [social-post]
secrets:
  ANTHROPIC_API_KEY: anthropic-key
  LINKEDIN_API_KEY: linkedin-key
```

### Apply Flow

```
zund apply --preview
  → reads agent secret mappings
  → collects requirements from role + skills
  → decrypts fleet/secrets/keys.yaml (SOPS + age key)
  → validates: all required secrets mapped AND exist
  → shows:
      ✓ ANTHROPIC_API_KEY → anthropic-key (set)
      ✓ LINKEDIN_API_KEY → linkedin-key (set)
      ⊘ IMAGE_GEN_API_KEY (not mapped, optional)

zund apply
  → same + injects decrypted values as container env vars
  → plaintext never written to disk, only held in memory during apply
```

### Remote Server / CI Setup

```bash
# New server — paste the age key once
zund setup
#   "Paste your age key:"
#   > AGE-SECRET-KEY-1QFNZ...
#   "Saved. Ready to apply."

# CI/CD — env var
export SOPS_AGE_KEY="AGE-SECRET-KEY-1QFNZ..."
```

### Upgrade Path (no zundd code changes)

```yaml
# fleet/.sops.yaml — change backend, run zund secret rotate
# age → AWS KMS:
creation_rules:
  - kms: "arn:aws:kms:us-east-1:123:key/my-key"
# age → Vault:
creation_rules:
  - hc_vault_transit_uri: "https://vault.company.com/v1/transit/keys/zund"
```

Env vars as fallback for quick starts: `secrets: { API_KEY: ${ANTHROPIC_API_KEY} }`

## 4. zundd Configuration

zundd itself is configured separately from fleet YAML. Fleet YAML is portable ("what I want"). zundd config is environment-specific ("where I'm running").

```yaml
# ~/.zund/config.yaml (or /etc/zund/config.yaml, or --config flag)

# Incus connection
incus:
  socket: /var/lib/incus/unix.socket     # Linux default
  # socket: /Users/nix/.colima/incus/incus.sock  # macOS + Colima
  project: zund                           # Incus project for fleet isolation

# Container defaults
containers:
  defaultImage: zund/base                 # base image for agents

# Data root — all managed data lives under this directory
data:
  dir: ~/.zund/data                       # root for all agent/fleet data

# API server
api:
  unixSocket: ~/.zund/zundd.sock          # CLI connects here
  tcp:
    enabled: true
    host: 0.0.0.0
    port: 4000                            # web console + remote access

# Logging
log:
  level: info                             # debug | info | warn | error
```

zundd config is **not** part of the fleet. It describes the environment — Incus socket path, ports, storage directories. Different machines running the same fleet YAML will have different zundd configs.

### Managed Data

zundd auto-creates and auto-mounts all per-agent data. Users don't configure paths — everything lives under `data.dir`.

```
~/.zund/data/
  sessions/
    writer/                  # Pi session JSONL files
      session-abc.jsonl
    reviewer/
      session-def.jsonl
  memory.db                  # single SQLite (facts + working memory, scoped by row)
  workspace/
    writer/                  # persistent workspace per agent
    reviewer/
  extensions/                # Pi extensions (shared across agents)
    zund-fleet.ts
  skills/
    brand-voice/             # local — copied from fleet dir on apply
      SKILL.md
      references/
    fleet-tools/             # builtin — shipped with zundd
    social-post/             # git — fetched and cached
      SKILL.md
      resources/
  _cache/
    git/                     # raw git clones (shared, deduped)
      github.com/company/agent-skills/
    registry/                # registry downloads (versioned)
      company/deploy-to-vercel/
        3.0.0/
        2.1.0/               # old versions kept until pruned
  state/
    fleet.json               # zundd internal state (container IDs, health)
```

**What zundd auto-mounts into every container (no user config):**

| Host path | Container path | Purpose |
|-----------|---------------|---------|
| `~/.zund/data/sessions/{agent}/` | `/root/.pi/agent/sessions` | Pi session persistence |
| `~/.zund/data/workspace/{agent}/` | `/workspace` | Persistent agent workspace |
| `~/.zund/data/extensions/` | `/root/.pi/agent/extensions` | Zund fleet extension |
| `~/.zund/data/skills/{name}/` | `/skills/{name}/` | Per-agent, only assigned skills (readonly) |

**How skills are provisioned:**

| Source | On apply | Stored at |
|--------|----------|-----------|
| `local` | Copied from fleet dir | `skills/{name}/` |
| `builtin` | Already present | `skills/{name}/` |
| `git` | Cloned to `_cache/git/`, extracted | `skills/{name}/` |
| `registry` | Downloaded to `_cache/registry/`, extracted | `skills/{name}/` |

Only skills assigned to an agent are mounted into that agent's container. An agent with `skills: [brand-voice, social-post]` gets two readonly mounts, not the whole skills directory.

**Memory is NOT mounted** — it runs inside the zundd process. Pi agents call memory tools via the `zund-fleet` extension, which calls back to zundd's API. A single `memory.db` holds all agents' facts and working memory, scoped by `(agent, scope)` rows:

```
Pi (in container) → memory_save("client prefers informal tone", scope: "agent")
  → extension → POST /v1/agents/writer/memory
  → row: (agent="writer", scope="agent:writer", content=...)

Pi (in container) → memory_search("client tone")
  → extension → GET /v1/agents/writer/memory?q=...&scope=agent:writer
  → zundd queries facts WHERE scope = 'agent:writer' → private results only
```

**Working memory** has three scopes — `agent` (persistent), `thread:{sessionId}` (session-local, GC'd on new session), and `fleet` (shared read-only, seeded from `kind: fleet` `workingMemory` field). All three are fetched in parallel on every turn by `before_agent_start` and appended to the agent's `systemPrompt` under three labeled sections. See `packages/daemon/ARCHITECTURE.md` for the full shape.

**User-configured mounts are for extra data only:**

```yaml
# Agent YAML — mounts field is optional, for additional data
mounts:
  - name: brand-assets
    host: /shared/brand-assets
    readonly: true
# mounted at /data/brand-assets in container
```

**Zero-config experience:** create an agent, apply, it gets sessions + workspace + extensions + memory automatically. No paths to configure.

## 4. How zundd Works

### Startup

```
zundd starts
  → Reads config (fleet folder path, ports, Incus socket)
  → Connects to Incus via Unix socket
  → Loads persisted fleet state from disk
  → Starts API server (Unix socket + TCP)
  → Ready — does NOT auto-apply, waits for commands
```

### Apply Flow

Apply accepts resources from two sources: files on disk, or inline payload. Both go through the same preview → confirm → execute pipeline.

**From files (CLI, manual edits):**
```
POST /v1/apply { preview: true, source: "file", path: "fleet/" }
  → Read all YAML from fleet folder
  → Parse + validate (Zod schemas)
  → Diff against running state
  → Return planned changes (no action taken)

POST /v1/apply { source: "file", path: "fleet/" }
  → Same diff → execute
  → For each CREATE: build container from zund/base image, write Pi config, start Pi
  → For each UPDATE: stop agent, update config, restart
  → For each DESTROY: stop container, cleanup
  → Persist new state to disk
  → Return results
```

**From payload (UI, programmatic — no file exists yet):**
```
POST /v1/apply { preview: true, resources: [{kind: "agent", name: "test", ...}] }
  → Merge payload resources with existing fleet state
  → Diff against running state
  → Return planned changes (no file written, no action taken)

POST /v1/apply { resources: [...], writeFiles: true }
  → Same diff → execute
  → Write YAML files to fleet folder
  → Create/update/destroy containers
  → Persist new state to disk
  → Return results
```

**CLI convenience commands (sugar over apply):**
```bash
zund apply -f fleet/ --preview         # preview files vs running state
zund apply -f fleet/                   # execute

zund agent add test --model claude-sonnet --preview   # preview inline, no file yet
zund agent add test --model claude-sonnet             # write file + apply

zund agent remove test                 # delete file + apply (stops agent)
```

### Pull Flow

```
GET /v1/fleet/export
  → Serialize running fleet state as YAML files
  → Client writes them to local fleet folder
  → User edits locally → preview → apply back
```

### Revert

No built-in undo. Revert is done through the same apply mechanism:

```bash
# Git-backed fleet folder (recommended)
git revert HEAD && zund apply -f fleet/

# No git — re-export running state to overwrite local files
zund pull -o fleet/

# Remove a specific agent
zund agent remove test                 # deletes file + stops agent
```

### Runtime (no file changes)

```
POST /v1/agents/:name/message    → proxy to Pi RPC → SSE stream back
GET  /v1/agents/:name/state      → agent status, model, message count
GET  /v1/agents                   → list all agents
GET  /v1/fleet/status             → fleet overview
```

## 4. Architecture Layers

```
Layer 1: Library (YAML)      — skills, tools definitions
Layer 2: Fleet Definition    — agents, teams, relationships (YAML files)
Layer 3: Runtime (API)       — state, conversations, sessions, events
Layer 4: Observation         — logs, health, dashboard, dev harness
```

YAML owns structure (layers 1-2). API owns runtime (layers 3-4). `apply` bridges them.

## 5. Key Technical Decisions (from experiments)

| Area              | Decision                                          | Evidence                                            |
| ----------------- | ------------------------------------------------- | --------------------------------------------------- |
| Incus transport   | Bun native `fetch` with `{ unix: socketPath }`    | POC 01 — undici not needed                          |
| Container startup | Pre-built `zund/base` image (407MB)               | POC 13 — 2.8s vs 150s cold start                    |
| Pi communication  | Long-lived `incus exec` process, stdin/stdout RPC | POC 04 — <1ms proxy overhead                        |
| Session storage   | Host-mounted dir with `shift=true`                | POC 02 — survives container restart                 |
| Ephemeral agents  | `incus copy --ephemeral` from template            | POC 06 — 228ms trigger-to-ready                     |
| Web UI            | Custom build, no Pi ChatPanel bridge              | POC 10A/10B — bridge too coupled to Pi internals    |
| Memory            | `@mastra/memory` + `@mastra/libsql` standalone    | POC 05, 09 — zero infra, works in Bun               |
| Server            | Two `Bun.serve` instances (Unix + TCP)            | POC 07 — shared state via closure                   |
| Pi extensions     | `.ts` files in extensions dir, loaded via jiti    | POC 08 — tools, context injection work              |
| Pi sessions       | JSONL v3, parseable without SDK                   | POC 11 — file not written until first assistant msg |

## 6. Build Slices

Each slice ends with something verifiable. Tests are written alongside, not after.

### Slice 1: zundd boots, harness shows it's alive

- Scaffold `packages/daemon` (tsconfig, package.json, entry point)
- `Bun.serve` dual (Unix socket + TCP)
- Routes: `GET /health`, `GET /v1/fleet/status`
- Dev harness at `:9999` (connects to zundd, shows status)
- Unit tests

**Verify:** start zundd, open harness, see "connected, no agents"

### Slice 2: zundd reads YAML, preview shows what will happen

- Fleet folder config (point zundd at a directory)
- YAML parser (agent + skill definitions, Zod validated)
- `POST /v1/apply { preview: true }` → returns planned changes
- Dev harness shows parsed config + preview diff
- Unit tests for parsing + diffing

**Verify:** write agent YAML, preview in harness shows "+ content-writer CREATE"

### Slice 3: apply launches an agent, I can message it

- IncusClient (create, start, exec, stop, delete)
- Image check (`zund/base` exists?)
- Pi launcher (from image, RPC mode)
- `POST /v1/apply` → actually creates containers + starts Pi
- `POST /v1/agents/:name/message` → proxy to Pi → SSE stream
- `GET /v1/agents/:name/state`
- Dev harness: running agents, live event stream, test message input
- Integration tests

**Verify:** apply, send message via harness, see streaming response

### Slice 4: full lifecycle — update, destroy, pull

- Apply handles updates (model change → restart)
- Apply handles destroy (agent removed from YAML → stop + cleanup)
- `GET /v1/fleet/export` → pull running config as YAML
- Session persistence (host mounts)
- Health checks
- Dev harness: full lifecycle + session info

**Verify:** add → apply, modify → apply, remove → apply, pull → get files back

### Slice 5: web console

- `packages/console` (from 10B learnings)
- Chat with agents (streaming SSE)
- Fleet overview (agents, status, health)
- Create/edit/delete agents → shows YAML + preview → apply

**Verify:** full workflow in browser

### Slice 6: skills, tools, memory

- Skills/tools YAML library
- Pi extension (fleet context injection)
- Memory engine (Mastra) + memory tools

**Verify:** define skill in YAML, assign to agent, agent uses it. Remember + recall works.

### Slice 7: CLI

- `packages/cli` — thin HTTP wrapper
- `zund init`, `zund start`, `zund stop`
- `zund apply --preview`, `zund apply`
- `zund pull`
- `zund agent message`, `zund status`

**Verify:** entire workflow from terminal

### Slice 8: artifacts

- `ArtifactPolicy` type + `artifacts` table in `memory.db` (content-addressed, SHA-256 IDs)
- `ArtifactStore` interface + `LocalArtifactStore`; REST endpoints (`POST/GET/DELETE /v1/artifacts`)
- `zund_emit_artifact` Pi tool registered in `extension.ts`; `/artifacts` bind-mount per agent container
- CLI `artifact list|get|rm|open` subcommands; fix stale event names in `agent chat`/`message`
- Console `AttachmentRenderer` — inline audio/image/video/PDF/markdown/HTML; download fallback for office formats
- TTL sweeper (30min interval, dedup-safe blob GC); smoke test asserts narrator audio artifact round-trip

**Verify:** `scripts/smoke-test.sh` with `ELEVENLABS_API_KEY` — narrator emits MP3, CLI prints download path, console renders `<audio>` element. See `docs/slice-7-plan.md` for full detail.

## 7. Testing Strategy

```
packages/daemon/test/
  unit/              bun test, pure logic, no infra, fast
  integration/       bun test, needs running zundd + Incus
  harness/           live web dashboard at localhost:9999
```

**Unit tests:** Every module gets tests alongside it. Parser, differ, config loader, SSE encoder — all pure logic, no mocks of infra.

**Integration tests:** Scripted scenarios against live zundd. Create → message → verify → destroy. Run manually or in CI with Incus available.

**Dev harness:** Lightweight web page at `:9999`. Connects to zundd's API (same endpoints users hit). Shows running agents, live event stream, apply preview. Has scenario runner for scripted flows. No extra deps — vanilla HTML + SSE + fetch. Built in Slice 1, grows with each slice.

---

## 8. Operating Model

- Work in vertical slices — each ends with something the user can verify
- Every module gets unit tests as it's built
- Dev harness is always open during development (terminal 1: zundd, terminal 2: harness in browser)
- Experiments are reference only — don't import, don't modify
- Fleet folder is files. UI writes files. CLI writes files. Manual vim writes files. Apply reads them.
