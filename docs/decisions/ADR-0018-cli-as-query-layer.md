# Read commands are the query API — prefer the CLI over raw `.index.json`

## Context

ADR-0005 established the two-layer split: source frontmatter stays minimal; derived fields live in a committed `docs/.index.json` regenerated by `docops index`. The ADR's "consequences" note already hinted that agents should prefer CLI read commands for derived fields, but it did not commit to that as the primary query mechanism.

Today's 30-document repo produces a ~30 KB index. Projecting a mature phase-2 repo (roughly 50 CTX + 200 ADR + 500 TP), the index will be 1–2 MB. That is still readable into a 200 k-token context window, but it is wasteful: almost no real query needs the full graph at once, and every agent session that blindly loads `.index.json` burns context on docs it will not use.

The absence of focused read commands (`get`, `list`, `graph`, `next`) forces agents to either read raw `.md` files or load the whole index. Neither is right. We need to close the surface and make the preference explicit.

## Decision

The canonical way to query a DocOps repo is through the CLI. Every agent-facing read returns a focused slice — never the whole index unless explicitly requested.

### Phase-1 read surface

| Command | Returns |
|---|---|
| `docops state` | compact snapshot — counts, needs-attention, active work, recent activity (TP-005, shipped) |
| `docops audit` | only the findings (TP-006, shipped) |
| `docops validate` | only the errors / warnings (TP-003, shipped) |
| `docops index [--json]` | writes the full index; `--json` streams it to stdout for bootstrap consumers and CI only (TP-004, shipped) |
| `docops get <id> [--json]` | one `IndexedDoc` by ID — source + derived fields + reverse edges |
| `docops list [--kind …] [--status …] [--tag …] [--json]` | trimmed listing with the frontmatter + reverse-edge fields agents actually need |
| `docops graph <id> [--depth N]` | typed edges outward from a starting doc, one or N hops |
| `docops next [--assignee …]` | single recommended task (lowest-priority unblocked backlog/active) |
| `docops search <query> …` | content + structured filter matches (ADR-0017) |

`get`, `list`, `graph`, and `next` are implemented by TP-012.

### `.index.json` is not the primary API

The committed `.index.json` exists for three consumers, in priority order:

1. Bootstrap / CI jobs that need the whole graph once (offline diffs, freshness checks).
2. Power users / scripts that need an ad-hoc query not expressible via the CLI (rare; we add a CLI flag instead when it recurs).
3. The CLI itself when it is deciding what to emit.

Agents running in a DocOps repo should reach for the CLI read commands first. The AGENTS.md template (what `docops init` writes into user repos) is updated to say so explicitly.

### Scaling story

If a repo's `.index.json` grows past a pain point (a rough threshold of ~2 MB is enough to matter for humans reading diffs; agent context budgets set the real limit), the project will:

1. Shard the index per kind (`.docops/index/context.json`, `.../decisions.json`, `.../tasks.json`). Each CLI read command picks the right shard.
2. Eventually switch to a SQLite-backed index for projects where streaming is cheaper than re-parsing.

Neither is built now. This ADR commits to keeping the option open by making every read go through the CLI — when we change the storage, callers don't notice.

## Rationale

- Keeps agent context focused. A `docops get ADR-0012` returns ~1 KB; a whole-index load is 30 KB–2 MB.
- Preserves the single-source-of-truth promise without tying agents to a specific file format.
- Matches the UNIX-y CLI-first commitment (ADR-0011).
- Unblocks future storage changes (sharding, SQLite) without rewriting every consumer.

## Consequences

- TP-012 ships the missing read commands. Until it does, agents legitimately still read `.index.json` directly for those queries; the AGENTS template is honest about the current gap.
- `docops.yaml` may later gain a flag for committing or not committing `.index.json` (per ADR-0005); default remains "commit it" for offline readability.
- Semantic search (ADR-0017 addendum) remains out of scope and would require its own ADR.
- Projects that need a custom read can propose a new CLI subcommand via an ADR + task rather than shipping one-off scripts against `.index.json`.
